# Управление памятью и контекстом ИИ-агента

## Задача

Wiki растёт с каждым источником, но LLM-агент с ограниченным контекстным окном выгорает при росте базы. Каждое чтение `index.md` + 3–5 страниц = 25K+ токенов до ответа (~2K). Без управления контекстом агент теряет состояние между вызовами и накапливает фантомный мусор в сессии.

**Цель**: Внедрить механизмы, которые минимизируют token burn и предотвращают деградацию синтеза данных — через файл-мост `working_memory.json`, CONTEXT_BUBBLE, Grep-контракт и скриптовую автоматизацию.

## Приёмы управления памятью

| Код | Что делает | Когда использовать |
|-----|-----------|-------------------|
| **CONTEXT_BUBBLE** | Ограничение: не более 3 активных страниц в контексте одновременно | Всегда при чтении wiki |
| **FLUSH_RAWS** | После извлечения данных из raw/ — забыть содержимое, никогда больше не читать | В конце ingest-цикла |
| **VIRTUAL_LOAD** | Перед вызовом grep/cat сформулировать цель: «Зачем мне эти данные?» Если размыто → отмена операции | Перед любым чтением файла |
| **Delta-Scoping** | Графовое соседство: читать только целевую страницу + 1-й и 2-й круг связей, не всю директорию | При поиске по wiki |
| **Compaction** | Сжать прочитанный контекст в саммари → записать в working_memory.json | В конце query (не discussion) |
| **Dismiss** | Выгрузить страницы из активного окна; следующий вызок начинается с self-memory.json | После ответа на query |
| **Topic-Switching** | Явный маркер `[DISCUSSION ENDED: topic]` → выгрузить предыдущий контекст | При смене темы обсуждения |

## [2026-06-25] Обсуждение: механизм Dismiss & Compact после query

### Проблема
После query-сессии агент держит в контексте: `index.md` (500+ строк), 3–5 полных wiki-страниц, все результаты grep/semantic search, JSON-инструкции процесса. Ответ занимает ~2K токенов. Всё остальное — фантомный контекст, который мешает следующим вызовам.

### Решение: Compaction после synthesis
В конце `process-query.json` добавить шаг 2.7 "context_compaction_and_dismissal": сжать прочитанный контекст в структурированное саммари (максимум 2K токенов), записать в `working_memory.json`, выгрузить raw-страницы из активного окна.

### Адаптивная политика: policy-driven dismissal
Разные задачи требуют разных политик удержания:
- **Query-mode**: агрессивная очистка после ответа (compaction=true, dismiss_after_response=true)
- **Project-mode**: частичная компактизация — выгружать tool results, но держать project state
- **Discussion-mode**: удерживать всё до завершения темы (compaction=false, keep=[topic_context, related_concepts])

Режим определяется явным маркером пользователя: `[WORK_MODE: query]` или `[WORK_MODE: discussion]`.

### Токеновая экономика
| Фаза | Токенов в контексте | Что держим |
|------|---------------------|------------|
| Search | ~15K–20K | index.md + semantic results + grep |
| Synthesis | ~25K–35K | index.md + 3-5 страниц + инструкции процесса |
| Compaction (до) | ~25K–35K | Всё ещё держим всё |
| Compaction (после) | ~1K–2K | Только саммари в working_memory.json |

### Архитектурные маркеры памяти
- `[ARCHIVE_STATE]` — старая история диалога, агент не должен перечитывать. Периодически сжимать в одностраничное саммари.
- `[WORKING_MEMORY]` (Оперативная память) — блок, который агент обязан читать и перезаписывать при каждом цикле. Содержимое: Focus, список файлов, Dead ends.
- `[HOVER_CONTEXT]` (Временный фокус) — динамический блок, куда подгружается контент только тех страниц, которые изменяются прямо сейчас.

### Скриптовая автоматизация вместо LLM-чтения
- Умный поиск: локальный скрипт FTS5 по wiki вместо «прочитай index.md». Агент получает 3 релевантных абзаца.
- Скриптовый линтинг: поиск сиротских страниц, пустых ссылок — `os.walk` или bash-скрипт. Агент получает сухой остаток.

## [2026-06-25] `working_memory.json` — файл-мост между сессиями

### Проблема: чистое окно контекста при каждом вызове

**Ситуация**: LLM-агент не сохраняет состояние между вызовами. Каждый новый запрос — чистый контекст (zero state). При этом:
1. **Agent теряет подзадачу** — если ingest был последним действием, агент не знает, что делать дальше.
2. **Agent ходит по кругу** — без dead_ends tracking, он может снова попробовать тот же grep-подход, который уже провалился.
3. **Токенная нагрузка на index.md** — при росте wiki (100+ страниц) agent читает index.md целиком (~500 строк), чтобы понять контекст.
4. **Нет моста между сессиями** — если agent сделал ingest, но сессия завершилась, следующий вызок начинается заново: нужно снова читать index.md, искать страницы для обновления.

### Решение: `working_memory.json` как мост между сессиями

Файл-мост, который:
1. **Агент читает при старте** — узнаёт, на чём остановился прошлый запуск.
2. **Агент перезаписывает при завершении** — сохраняет фокус, dead ends, next steps.
3. **Не дублирует wiki** — хранит только *метаданные* сессии (фокус, состояние), не сами страницы.

### Какие процессы будут использовать working_memory.json?

| Процесс | Когда читает | Что записывает | Как использует |
|---------|-------------|----------------|----------------|
| **Query** | При старте — `focus_node` может быть `"project_alpha"`, `open_pages` = `["wiki/entities/..."]` | `query_summary`, `next_steps_todo` (если найден novel insight → flag для создания страницы) |
| **Ingest** | При старте — понимает, какие entity/concept страницы нужно обновить. | `focus_node = "updated: pi-coding-agent.md"`, `open_pages = [...]`, `dead_ends += ["grep по всем wiki"]` |
| **Discussion** | При продолжении дискуссии — `query_summary` содержит предыдущие аргументы, `open_pages` = все связанные страницы. | Удерживает всю тему (compaction=false), добавляет новые аргументы в `query_summary` |
| **Project** | При итеративной работе — `focus_node` = текущая задача, `open_pages` = wiki-страницы + рабочие файлы проекта. | Обновляет `next_steps_todo`, `dead_ends` для текущего проекта. |
| **Lint** | Получает отчёт из bash/Python скрипта → записывает проблемы в `next_steps_todo`. | Не читает wiki напрямую — только получает данные от скрипта. |

## [2026-06-25] Финальная структура working_memory.json

### Решения по структуре:

1. **Session ID** — ❌ убрано, достаточно `last_updated` как идентификатора сессии.
2. **Query summary** — ✅ только когда нужен (query/discussion/project). Для ingest/lint — опционально или не используется.
3. **Dead ends** — ✅ JSON-массив объектов `{approach, reason}`. Позволяет агенту читать контекст неудачных подходов без повторения ошибок.
4. **Open pages** — статусы: `reading` (агент читает сейчас), `updating` (агент обновляет страницу), `read` (прочитана, можно выгрузить).

### Финальная схема

```json
{
  "last_updated": "YYYY-MM-DDTHH:MM:SS+TZ",
  
  "current_mode": "query | ingest | discussion | project | lint",
  "focus_node": "[[page_name]] — что агент делает прямо сейчас",
  
  "open_pages": [
    {"path": "wiki/entities/pi-coding-agent.md", "status": "reading" | "updating"},
    {"path": "wiki/concepts/llm-wiki-pattern.md", "status": "read"}
  ],
  
  "dead_ends": [
    {
      "approach": "grep по всем wiki для entity поиска",
      "reason": "слишком много шума, index.md работает лучше"
    }
  ],
  
  "query_summary": {
    "intent": "Что искал пользователь в последнем query",
    "pages_read": ["список прочитанных страниц"],
    "key_findings": ["3-5 пунктов"]
  },
  
  "next_steps_todo": [
    {"task": "Создать synthesis по RAG vs LLM Wiki Pattern", "priority": "high"}
  ]
}
```

### Режим работы: как агент определяет [WORK_MODE: project]

Агент **не ждёт явного маркера** — он сам определяет режим по контексту задачи. Три пути:

| Способ | Когда срабатывает | Пример |
|--------|------------------|--------|
| **Прямая инструкция пользователя** | Пользователь явно говорит «работаю над проектом» или ставит маркер | `[WORK_MODE: project]` в запросе |
| **Контекст задачи** | Агент понимает по формулировке, что это итеративная работа | «Мигрирую auth на JWT → какие схемы token rotation? → готово → payment integration» |
| **Уточняющий вопрос** | Агент не уверен → спрашивает пользователя | «Это одноразовый вопрос или часть проекта?» |

### Как агент управляет признаками режима:

1. Если агент определяет проект (итеративная работа с wiki как источником контекста):
   - Ставит `current_mode = "project"` в `working_memory.json`
   - Обновляет `wiki/snapshot.md` — добавляет проект в Active Projects
   - Подгружает snapshot.md при следующих вызовах
2. Если пользователь ставит явный маркер `[WORK_MODE: project]` → принимает немедленно.
3. Если контекст неоднозначен → уточняет у пользователя.

**Правило**: агент сам управляет режимом работы — гибкий механизм, основанный на контексте, прямой инструкции или уточнении.

### working_memory.json vs wiki/snapshot.md

| Уровень | Файл | Назначение | Когда читается |
|---------|------|------------|----------------|
| **Оперативный** | `working_memory.json` | Файл-мост между сессиями, фокус текущей задачи | При каждом вызове агента |
| **Информативный** | `wiki/snapshot.md` | Перечень активных проектов и связанных wiki-страниц | Только при `[WORK_MODE: project]` + Active Projects |

Оба файла решают одну задачу — оптимизация памяти без потери важного контекста, но на разных уровнях:
- working_memory.json = *как* агент работает прямо сейчас (оперативный фокус)
- wiki/snapshot.md = *какой* проект ведёт пользователь (информативный контекст)
