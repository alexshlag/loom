# Wiki Structure Guide

This document describes how the wiki is organized, how pages are structured, and the conventions you (or your agent) should follow when creating or modifying content.

---

## Overview: Directory Layout

The wiki lives in `wiki/` and is organized by **category**, not flat:

```
wiki/
├── entities/       ← Concrete objects: people, companies, technologies
│   └── <project>-<entity>.md         (e.g., symfony-messenger.md)
├── concepts/       ← Abstract ideas, principles, methodologies
│   ├── abstract-only.md              (truly framework-agnostic)
│   └── <project>-<concept>.md        (framework-specific concept)
├── syntheses/      ← Deep analysis combining ≥3 sources/pages
│   └── topic-analysis.md
├── comparisons/    ← Comparative analysis of related entities/concepts
│   └── <entity-a>-vs-<entity-b>.md
├── docs/           ← Long-form technical documentation, user manuals
│   ├── <project>-overview.md         (framework overviews)
│   └── <tool>-commands.md            (CLI reference guides)
├── assets/
│   ├── images/          ← Original image copies (.png, .jpg, .jpeg, .gif)
│   └── descriptions/    ← Markdown descriptions: OCR + entities + metadata
├── templates/       ← JSON template files for new pages (see below)
├── skills/          ← Agent skill definitions (auto-generated from trajectories)
├── index.md         ← Structured index with summaries per category
├── overview.md      ← One-page snapshot of all wiki facts
├── log.md           ← Chronological action journal
└── hot.md           ← Session context: focus node, open pages, next steps
```

> **Never manually edit `index.md`, `overview.md`, `log.md`** — they are auto-generated by scripts. Only the agent modifies them via `rebuild-meta.sh`.

## Categories & Routing

Categories are defined in [`rules/categories.json`](../rules/categories.json). The agent reads this file on every page creation — never hardcodes category definitions.

| Category | Purpose | Path Pattern | Example |
|----------|---------|-------------|---------|
| **entities** | Concrete objects (people, companies, technologies) | `<project>-<type>.md` | `symfony-messenger.md`, `nodejs.md` |
| **concepts** | Abstract ideas, methodologies | Abstract: bare name. Project-specific: `<project>-<name>.md` | `cache-system.md` (abstract), `symfony-messaging-pattern.md` (specific) |
| **syntheses** | Deep analysis from ≥3 sources/related pages | Topic-based | `rag-vs-llm-wiki-pattern.md` |
| **comparisons** | Side-by-side comparison of related entities | `<a>-vs-<b>.md` | `ai-factory-vs-pi.md`, `symfony-vs-laravel-messaging.md` |
| **docs** | Long-form documentation, guides, references | `<project>-<doc-type>.md` | `ai-factory.md`, `symfony-cli-guide.md` |

> **Rule:** Agent reads `categories.json` via schema_ref on every page creation. New categories added to this file automatically propagate through auto-crosslink routing (no hardcoded paths needed).

## Universal Frontmatter Schema

Every wiki page has YAML frontmatter at the top:

```yaml
---
tags: [tag1, tag2, tag3]              # 3-7 domain-specific keywords — see rules/tag-guidelines.json
date: YYYY-MM-DD                       # Current system date (never source-derived!)
type: documentation                    # documentation | code_reality | live_state
category: entities                     # Must match one of the categories above
aliases: [synonym1, synonym2]          # Discoverability synonyms — see rules/tag-guidelines.json#aliases_system
sources:                               # Data origins (raw/ paths, wiki relative paths, or "web_search")
  - raw/sources/SRC-2026-07-08-001/    # Raw source directory
  - wiki/entities/symfony-messenger.md # Related wiki page
  - web_search                         # Web search result (no path)
related:                               # Wiki-relative paths to related pages — empty = no connections
  - wiki/concepts/cache-system.md
---
```

### Field Details

| Field | Required | Description |
|-------|----------|-------------|
| `tags` | ✅ Mandatory | Domain-specific keywords (3-7). Never use generic tags like "tech" or "important". See [`rules/tag-guidelines.json`](../rules/tag-guidelines.md) for patterns and enforcement rules. |
| `date` | ✅ Mandatory | **Current system date** — YYYY-MM-DD format. Never derive from source filename, commit dates, or web crawl timestamps. |
| `type` | ✅ Mandatory | Reality layer: `documentation` (conceptual/historical), `code_reality` (actual implementation/code), `live_state` (current system state). Used in contradiction resolution priority cascade. |
| `category` | ✅ Mandatory | Must match one of the categories from `rules/categories.json`. Determines which directory page lives in and auto-crosslink routing. |
| `aliases` | ⚠️ Recommended | Discoverability synonyms for pages with multiple common names. Store variants — never create separate files. See [`rules/tag-guidelines.json#aliases_system`](../rules/tag-guidelines.md). |
| `sources` | ✅ Mandatory | Data origins: paths under `raw/`, wiki-relative paths (`wiki/entities/foo.md`), or `"web_search"` for web-derived content. At least one source required per page. |
| `related` | ⚠️ Recommended | Wiki-relative paths to connected pages. Empty array = no connections (rare). Agent auto-suggests via crosslink scoring algorithm. |

### Auto-computed Fields

- **evidence_grade**: Assigned automatically from source authority → [`rules/evidence_grade.json`](../rules/evidence_grade.md). Higher evidence grades win in contradiction resolution.

## Page Templates

Templates live in `wiki/templates/` as JSON files. They define recommended section structure for each category — not enforced rules, but strong guidance that the agent follows.

### Entity Template (`entity-template.json`)

Recommended sections:
1. **Definition** — What is this entity?
2. **Key Characteristics** — Important attributes, properties
3. **Architecture / Implementation Details** — How it's built (if applicable)
4. **Usage Examples** — Code snippets, command examples
5. **History / Evolution** — Version history, changes over time
6. **Related Pages** — Crosslinks to other wiki pages

### Concept Template (`concept-template.json`)

Recommended sections:
1. **Definition** — What is this concept?
2. **Core Principles / Architecture** — Foundational ideas
3. **How It Works** — Mechanics, patterns of application
4. **Examples / Application** — Real-world usage scenarios
5. **Related Pages** — Crosslinks to related concepts/entities

### Synthesis Template (`synthesis-template.json`)

Recommended sections:
1. **Context** — Why this analysis? What's the question being answered?
2. **Analysis** — Deep dive combining ≥3 sources into coherent argument
3. **Conclusions** — Summary of findings, implications

> **Syntheses are special:** They require ≥3 distinct sources and create new insights not available in any single source page.

### Comparison Template (`comparison-template.json`)

Recommended sections:
1. **Overview** — Brief intro to both items being compared
2. **Comparison Table** — Side-by-side feature matrix
3. **Deep Dive Analysis** — Detailed analysis of differences/similarities
4. **Conclusions / Recommendation** — Which is better for what use case?
5. **Related Pages** — Crosslinks to entity/concept pages

### Docs Template (`docs-template.json`)

Recommended sections:
1. **Overview** — Brief intro paragraph defining scope and purpose
2. **Prerequisites / Setup** — Runtime versions, tools needed, environment config
3. **Getting Started** — Step-by-step first run instructions
4. **API Reference** — Endpoints/commands grouped by resource with request/response examples
5. **Configuration** — Environment variables, config files, feature flags
6. **Deployment** — Build steps, CI/CD, monitoring (if applicable)
7. **Migration Guides** — Version upgrades, breaking changes (when relevant)
8. **See Also** — Crosslinks to wiki/entities/ and wiki/concepts/

#### Docs Navigation Pattern

Docs pages use a navigation header/footer:

```markdown
[← Previous Topic](prev.md) · [Back to Index](../docs-index.md) · [Next Topic →](next.md)
```

- First docs page has only "Back to Index" link
- Last docs page has only "← Previous Topic" link
- Middle pages have all three links

> **Mandatory:** Navigation header required for all docs pages. `See Also` section required as footer.

## Naming Conventions

To prevent filename collision across projects/frameworks, all wiki pages must follow naming rules defined in [`rules/naming_conventions.json`](../rules/naming_conventions.md).

### General Rule: Project Prefixing

**Every page name is prefixed with the project/framework slug.** Bare concept names are only allowed for truly abstract concepts that apply universally.

| Directory | Pattern | Examples |
|-----------|---------|----------|
| `entities/` | `<project>-<type>.md` | `symfony-messenger.md`, `nodejs.md`, `ai-factory.md` |
| `concepts/` (abstract) | bare name | `cache-system.md`, `hexagonal-architecture.md` |
| `concepts/` (project-specific) | `<project>-<name>.md` | `symfony-messaging-pattern.md`, `react-hooks-pattern.md` |
| `docs/` | `<project>-<doc-type>.md` | `ai-factory.md`, `symfony-cli-guide.md` |
| `comparisons/` | `<a>-vs-<b>.md` | `ai-factory-vs-pi.md`, `symfony-vs-laravel-messaging.md` |

### Exception List

Truly abstract concepts that don't need prefixes:
- `cache-system.md` — Universal caching patterns (PSR-6, cache-invalidation)
- `hexagonal-architecture.md` — Clean architecture pattern (ports-and-adapters), universal across frameworks
- `doctrine-orm.md` — Doctrine ORM works with any PHP framework

### Detection Logic

The agent uses [`scripts/filename-audit.sh`](../scripts/filename-audit.md) to detect violations:
1. If tags contain project-name AND filename lacks prefix → **VIOLATION** (unless in exception list)
2. If related contains `entities/<project>.md` AND filename lacks prefix → **VIOLATION**

> **Auto-audit runs:** During lint workflow (check_id=6). Violations flagged as HIGH severity — require user approval before rename.

## Assets & Media Pipeline

### Image Ingest Flow

1. **Copy original image** to `raw/assets/images/<filename>.<ext>`
2. **Extract OCR + entities** via agent's vision capabilities → save description in `raw/assets/descriptions/<filename>.md`
3. **Wikilink** from wiki page: `[image](../assets/images/filename.png)` with caption
4. **Add backlink** to image description file: `related: [wiki/entities/xxx.md]`

### Image Description Format

```markdown
---
tags: [image, <context>]
date: YYYY-MM-DD
category: note
sources: [raw/assets/images/filename.png]
---

# Description: <Image Title>

## OCR / Content
<Extracted text from image via OCR>

## Entities Identified
- Entity 1 (with wiki crosslink)
- Entity 2 (with wiki crosslink)

## Metadata
- Dimensions: WxH pixels
- Format: PNG/JPG/GIF
- Date ingested: YYYY-MM-DD
```

## Memory & Context Files

These files are **not** part of the knowledge graph — they're session metadata that helps the agent maintain context across turns.

| File | Purpose | Lifetime | Read/Write |
|------|---------|----------|------------|
| `working_memory.json` | Bridge: focus_node, open_pages, dead_ends, next_steps_todo | Per-session (clear & rewrite on update) | Read at start, write after each operation |
| `wiki/hot.md` | Current session's active focus and decisions | Per-session | Read/written by agent during session |
| `wiki/log.md` | All actions recorded chronologically | Permanent | Append-only via `## [YYYY-MM-DD] action \| description` format |

> **Never read these files as knowledge** — they're operational metadata. The wiki's actual knowledge is in `entities/`, `concepts/`, etc.

## Quick Reference: Page Creation Checklist

When creating a new wiki page, follow this checklist:

- [ ] **Read categories.json** — confirm category exists and matches content
- [ ] **Generate frontmatter** — tags (3-7), date (today!), type, category, sources, related
- [ ] **Apply naming convention** — prefix with project slug unless abstract concept
- [ ] **Follow template structure** — use appropriate sections from wiki/templates/
- [ ] **Add crosslinks** — populate `related:` array and add wikilinks to body
- [ ] **Validate path** — run validate-path.sh guardrails before writing
- [ ] **Update index.md + log.md** — rebuild metadata via rebuild-meta.sh

> **Agent reads:** All rules via schema_ref from process files (process-ingest.json, process-query.json). Never hardcode rules in memory.
